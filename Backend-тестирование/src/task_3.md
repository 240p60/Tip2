# Задание 3. Базы данных

## 1. Теоретическая часть

### Реляционные vs нереляционные БД

| Критерий | Реляционные (SQL) | Нереляционные (NoSQL) |
|----------|-------------------|------------------------|
| Структура данных | Таблицы с фиксированной схемой, связи через FK | Документы / ключ-значение / графы / широкие колонки (без жёсткой схемы) |
| Язык запросов | SQL — единый стандарт | У каждой СУБД свой DSL/API (MongoDB Query Language, CQL, Redis-команды и т.п.) |
| Целостность | ACID-транзакции, FK, constraints | Часто eventual consistency, BASE-модель; ACID — частично (например, в MongoDB ≥ 4.0) |
| Гибкость схемы | Жёсткая, изменения через миграции | Гибкая, поле можно добавить «в один документ» без миграции всей коллекции |
| Масштабирование | Vertical (мощнее железо), sharding сложен | Horizontal (добавление нод), шардирование «из коробки» |
| Идеальные кейсы | Финансы, ERP, биллинг, любая бизнес-логика с транзакциями | Логи, ленты, сессии, кеши, real-time-аналитика, графы связей |
| Примеры СУБД | PostgreSQL, MySQL, Oracle, MS SQL Server, СУБД 1С | MongoDB, Cassandra, Redis, DynamoDB, Elasticsearch, Tarantool, ClickHouse |

**Плюсы реляционных:** надёжность, зрелый стек, ACID, мощные JOIN-операции и аналитика, понятные миграции.
**Минусы реляционных:** хуже горизонтальное масштабирование, жёсткая схема замедляет эволюцию продукта.

**Плюсы нереляционных:** легко скейлятся, гибкая схема, высокая производительность на простых операциях, удобны для big data.
**Минусы нереляционных:** слабее с транзакциями (исторически), нужно дублировать данные ради скорости, отчётность сложнее.

### MySQL vs PostgreSQL (две разные реляционные СУБД)

| Критерий | MySQL | PostgreSQL |
|----------|-------|------------|
| Тип | Open-source реляционная СУБД (Oracle) | Open-source объектно-реляционная СУБД (PGDG) |
| Лицензия | GPL/Commercial (Oracle) | PostgreSQL License (BSD-like) |
| Движки хранения | InnoDB (по умолчанию), MyISAM, MEMORY | Один движок, расширяемый через extensions |
| Типы данных | Стандартные + JSON | Стандартные + JSON/JSONB + массивы + geo (PostGIS) + кастомные типы |
| Соответствие стандартам SQL | Частично, со «своими» особенностями | Очень близко к стандарту ANSI SQL |
| Транзакции и MVCC | InnoDB поддерживает | Поддерживает, причём более «классическим» MVCC |
| Расширяемость | Ограниченная | Через extensions (`pg_stat_statements`, `pg_trgm`, `pgcrypto`, `PostGIS`, `TimescaleDB`) |
| Полнотекстовый поиск | Базовый (FULLTEXT в InnoDB) | Сильный встроенный, плюс расширения |
| Репликация | Statement / row-based | Streaming replication + logical replication |
| Производительность на простых OLTP | Часто быстрее, лёгкий вес | Сравнимо, но мощнее на сложных аналитических запросах |
| Типичное применение | Веб-приложения, CMS (WordPress, Joomla), быстрые OLTP | Сложные бизнес-приложения, BI, GIS, fintech, госы |

В итоге **MySQL** — «легче и быстрее на простых сценариях», **PostgreSQL** — «мощнее, расширяемее, ближе к стандарту». Обе — реляционные, обе используют SQL, но архитектурно очень разные.

## 3. Гипотеза о типе БД (без интернета, только на основе моих знаний)

Гипотеза: **гибридный подход** — VK с высокой вероятностью использует **несколько разных типов БД одновременно**, потому что у сервиса принципиально разная природа данных. «Чистый» реляционный или «чистый» nosql вариант на таком масштабе физически не выживет.

### Аргументация (от функциональности к архитектуре)

1. **Лента / посты / комментарии** — сотни миллионов записей в день. Здесь критична **скорость записи и горизонтальная масштабируемость**, целостность не критична на уровне отдельной записи (потерянный лайк не страшен). → подходит **нереляционная** БД (документная или key-value).
2. **Личные сообщения** — миллиарды сообщений, нужен быстрый доступ по диалогу, упорядоченность по времени. Здесь нужны или специализированные хранилища (типа «лента сообщений»), или собственные движки. → **нереляционная или специализированная in-memory** БД.
3. **Профили пользователей и связи (друзья, подписки)** — структурированные данные с жёсткими связями. С одной стороны, нужно ACID для биллинга/верификации. С другой — граф связей лучше ложится на графовую БД или собственный движок. → **гибрид: реляционная для основных данных + графовая/специализированная для связей**.
4. **Кэш и сессии** — короткоживущие пары ключ-значение для быстрой авторизации, антифрод-троттлинга и т.п. → **key-value in-memory (Redis-подобное хранилище)**.
5. **Поиск (по людям, видео, аудио)** — требует обратных индексов, ранжирования, fuzzy matching. → **поисковый движок** (Lucene-семейство / собственный).
6. **Биллинг (платная музыка, VK Pay, реклама)** — деньги, ACID, аудит. → строго **реляционная БД**.

### Итог гипотезы

VK почти наверняка использует **сильно гибридную архитектуру:** для каждой группы функциональности — своё хранилище. Основа — собственные in-memory движки для лент и мессенджера + классическая реляционная БД для биллинга и пользователей + key-value кэш-уровень. С большой вероятностью используется не «вендорная MongoDB/Cassandra», а **внутренние решения** (потому что VK исторически известен тем, что пишет своё под нагрузку).

## 4. Разбивка сервиса VK на 4 ключевые функции

| # | Функция | Какие данные | Тип БД (моё мнение) | Почему |
|---|---------|--------------|---------------------|--------|
| 1 | **Авторизация и сессии** | `user_id`, `session_token`, `device_id`, `ip`, `last_activity` | **Key-value in-memory** (Redis-подобное) | Огромное число коротких read/write-операций, данные хранятся миллисекунды-минуты, нет необходимости в JOIN, нужна молниеносная задержка на каждый запрос. |
| 2 | **Личные сообщения / чаты** | `message_id`, `chat_id`, `sender_id`, `text`, `attachments[]`, `created_at`, `read_at`, `forward_chain[]` | **Специализированная append-only БД** + кеш в памяти | Сообщения добавляются последовательно, удаляются редко. Нужна сортировка по времени в рамках чата, мгновенная доставка. Реляционная БД с миллиардами строк будет упираться в индекс на (`chat_id`, `created_at`). Лучше — log-structured хранилище или специализированный «message engine». |
| 3 | **Лента / посты / комментарии** | `post_id`, `author_id`, `community_id`, `text`, `attachments[]`, `likes_count`, `reposts_count`, `comments[]`, `published_at` | **Документная нереляционная БД** + поисковый индекс | Записи имеют разнородную структуру (пост со стикером ≠ пост с фото ≠ пост с опросом). Денормализация (хранить `likes_count` рядом) ускоряет чтение в ленте в десятки раз по сравнению с JOIN'ами. Поиск по тексту — отдельный поисковый движок. |
| 4 | **Биллинг и платежи (VK Pay, реклама, подписки)** | `transaction_id`, `user_id`, `amount`, `currency`, `provider`, `status`, `created_at`, `audit_log[]` | **Классическая реляционная БД (PostgreSQL-подобная)** | Здесь критичны ACID-транзакции, отчётность, аудит, FK-constraints. Любая потеря или дубликат строки = потеря денег и регуляторные риски. Гибкость схемы здесь не нужна — структура устаканивается. |

## 5. Идеи для тест-кейсов (по 1-2 на каждый тип БД)

### 5.1. Key-value in-memory (для сессий)

**TC-1. Истечение токена сессии**
- **Pre:** пользователь успешно логинится, получает `session_token` с TTL = 30 мин.
- **Steps:** дождаться 30 минут + 5 секунд (ускоренно — сдвинуть TTL в тест-окружении), выполнить запрос на любой защищённый endpoint с этим токеном.
- **Expected:** возврат 401 Unauthorized; токен отсутствует в Redis-key-value хранилище (`redis-cli GET session:<token>` → `nil`).
- **Зачем:** проверка корректной работы TTL — частый источник багов, особенно при сменах TZ.

**TC-2. Гонка при логине с двух устройств**
- **Pre:** пользователь логинится одновременно с двух устройств.
- **Steps:** два параллельных запроса POST `/auth/login` с одинаковыми credentials.
- **Expected:** оба возвращают валидные токены, оба токена существуют в KV-хранилище; обращение по любому из них валидно.
- **Зачем:** убедиться, что нет race condition с перезаписью ключа, и оба токена не «затирают» друг друга.

### 5.2. Документная нереляционная (для ленты)

**TC-3. Schema drift при добавлении нового типа поста**
- **Pre:** новая функция «опросы в посте» — поле `poll_options[]` добавлено в документ поста.
- **Steps:**
  1. Создать пост старого формата (без `poll_options`) и новый пост с `poll_options`.
  2. Загрузить ленту через API, открыв оба поста.
- **Expected:** старый пост без `poll_options` рендерится корректно (отсутствие поля не ломает фронт); новый пост рендерит опрос.
- **Зачем:** проверить «толерантность» документной модели к разным версиям схемы (это её главное преимущество — но и риск, если фронт жёстко ждёт поля).

**TC-4. Консистентность счётчиков лайков**
- **Pre:** на пост поставили 10 лайков.
- **Steps:** прочитать `likes_count` из документа поста и одновременно подсчитать число записей в коллекции `likes` для этого `post_id`.
- **Expected:** значения совпадают.
- **Зачем:** в денормализованных нереляционных БД счётчики обновляются отдельным процессом, и часто там возникают рассинхроны (eventual consistency «протекает» в UX).

### 5.3. Реляционная (для биллинга)

**TC-5. Атомарность транзакции при оплате**
- **Pre:** баланс пользователя = 100 ₽, оплата стоит 70 ₽.
- **Steps:** инициировать оплату; на середине операции (после списания, до коммита) принудительно убить процесс приложения (kill -9, или ввести искусственную ошибку через chaos-тест).
- **Expected:** после восстановления баланс пользователя = 100 ₽ (списания не произошло), транзакция в статусе `failed/rolled_back`, никаких «полу-провалившихся» записей в `payments`.
- **Зачем:** доказать, что используется реальная ACID-транзакция, а не «эмуляция».

**TC-6. Уникальность order_id и защита от дубликатов**
- **Steps:** одновременно отправить два запроса POST `/pay` с одним и тем же `order_id` (через два разных HTTP-клиента).
- **Expected:** только одна запись в таблице `payments` (FK + UNIQUE constraint на `order_id`); второй запрос возвращает 409 Conflict.
- **Зачем:** проверка, что бизнес-логика и БД-уровень согласованы и защищают от дублей платежей (классический баг при ретраях).

## 6. Проверка гипотезы

> *После выполнения задания проведи быстрый поиск в Интернете «[Название сервиса] tech stack database». Совпали ли твои гипотезы с реальной архитектурой?*

Известные публичные факты о технологическом стеке VK (по доступным источникам — статьи на Хабре, выступления инженеров VK на Highload++ и HolyJS):

- **VK действительно использует гибридный подход.** «Чистого SQL» или «чистого NoSQL» нет.
- В качестве **основного языка backend исторически был PHP** (форк под названием **KPHP** — собственный компилятор PHP в C++ от команды VK).
- **Собственные внутренние движки** (engines), написанные на C/C++ — для конкретных задач: ленты, кеша, текстового поиска, новостных потоков, голосовалок, рекомендаций. Один из публично-описанных — *Tarantool*-подобные in-memory движки.
- **Tarantool** — open-source in-memory СУБД, изначально разработанная в Mail.ru / VK (после объединения), активно используется внутри для горячих данных, очередей и кешей.
- **MySQL** — для классических реляционных данных (биллинг, метаданные пользователей в части кейсов).
- **ClickHouse** (создан в Яндексе, но активно используется и в VK) — для аналитики, логов и BI-задач.
- **Memcached / собственные in-memory кеши** — для горячих сессий и быстрого доступа.

### Сопоставление с моей гипотезой

| Что предположил | Что реально |
|------------------|-------------|
| Гибридный подход с несколькими типами БД | ✅ Подтверждено: PHP/KPHP + Tarantool + MySQL + ClickHouse + Memcached + собственные C++-движки |
| Key-value in-memory для сессий и горячих данных | ✅ Подтверждено: Memcached / Tarantool / Redis-like |
| Специализированный движок для ленты/сообщений | ✅ Подтверждено: собственные C++-движки + Tarantool |
| Реляционная БД (PostgreSQL-подобная) для биллинга | ⚠ Частично: используется MySQL, не PostgreSQL. По функциональному «классу» (реляционная ACID) — гипотеза верна, конкретно по СУБД ошибся в названии |
| Внутренние решения вместо «вендорной MongoDB/Cassandra» | ✅ Подтверждено: исторически VK много пишет своё (KPHP, engines), а внешние СУБД — выборочно |

### Вывод

Главный вывод гипотезы — «гибридный подход и несколько типов БД для разных задач» — подтвердился. Ошибся в детали: предположил, что биллинг будет на PostgreSQL, в реальности это MySQL. Это нормальная погрешность гипотезы: предметная область угадана верно, конкретный вендор — нет (это видно только из статей инженеров VK).

Для тестировщика главный практический вывод: при тестировании любого крупного сервиса нельзя считать, что вся данные хранятся «в одной БД». Тест-стратегия должна учитывать, что данные размазаны по 3-5 разным хранилищам, и баги могут возникать **именно на стыках** (например, в моменте, когда лента уже показывает новый лайк, а реляционная БД его ещё не зафиксировала, — типичный eventual-consistency-баг).
