# Задание 1. Клиент-серверная архитектура

## 1. Теоретическая часть

### Трёхзвенная архитектура (3-tier)

Классическая модель разделяет приложение на три логических слоя:

1. **Презентационный слой (Presentation tier / Client)** — то, что видит пользователь: HTML/CSS/JS в браузере или нативное мобильное приложение. Отвечает за отображение и сбор пользовательского ввода.
2. **Слой бизнес-логики (Application / Logic tier)** — backend-серверы, обрабатывающие запросы, реализующие правила, бизнес-процессы, валидацию, оркестрацию вызовов.
3. **Слой данных (Data tier)** — СУБД, файловые хранилища, кеши; отвечает за персистентное хранение и извлечение данных.

### Многозвенная архитектура (N-tier)

Дальнейшее развитие 3-tier, в котором каждый слой может быть разбит на несколько подсистем:

- между клиентом и логикой добавляются **CDN**, **API Gateway**, **Load Balancer**, **Reverse Proxy**;
- слой логики делится на **микросервисы**, **очереди сообщений (Kafka/RabbitMQ)**, **сервисы аутентификации (OAuth/SSO)**;
- слой данных — на **OLTP-БД**, **OLAP-хранилища**, **кеши (Redis/Memcached)**, **объектные хранилища (S3)**, **полнотекстовый поиск (Elasticsearch)**.

N-tier даёт лучшую масштабируемость и устойчивость, но усложняет интеграцию, наблюдаемость и тестирование.

## 2. Блок-схема архитектуры выбранного веб-приложения

**Выбранное приложение:** интернет-магазин **Ozon.ru** (e-commerce, многозвенная архитектура).

```
                       ┌─────────────────────────┐
                       │   Пользователь (UA)     │
                       │  Браузер / мобильное    │
                       │      приложение         │
                       └────────────┬────────────┘
                                    │ HTTPS
                                    ▼
                       ┌─────────────────────────┐
                       │           CDN           │   ← 1. Доставка статики, кеш у пользователя
                       │ (Cloudflare / Akamai)   │

                                    ▼
                       ┌─────────────────────────┐
                       │ Load Balancer / WAF     │   ← 2. Балансировка + защита от атак
                       │      (nginx / HAProxy)  │

   ┌──────────────┐        ┌──────────────┐        ┌──────────────┐
   │ Catalog svc  │        │ Cart svc     │        │ Payment svc  │   ← 4. Микросервисы бизнес-логики

   └──────┬───────┘        └──────┬───────┘        └──────┬───────┘
          │                       │                       │
          ▼                       ▼                       ▼
   ┌─────────────┐         ┌─────────────┐         ┌─────────────┐
   │ PostgreSQL  │         │ Redis cache │         │ Платёжный   │
   │ (товары)    │         │ + Postgres  │         │ шлюз        │   ← 5. Слой данных + внешние интеграции
   └─────────────┘         └─────────────┘         └─────────────┘
                                    │
                                    ▼
                          ┌──────────────────┐
                          │ Kafka (events)   │   ← 6. Шина событий между сервисами
```

### Ключевые компоненты и их назначение

| № | Компонент | Назначение |
|---|-----------|-----------|
| 1 | **CDN** | Раздача статики (картинки, JS, CSS) с ближайшего к пользователю узла. Снижает нагрузку на основной DC и латентность. |
| 2 | **Load Balancer + WAF** | Распределение трафика между серверами, защита от DDoS, SQL-инъекций и других web-атак. |
| 3 | **API Gateway** | Единая точка входа в backend, маршрутизация на нужные микросервисы, проверка токенов, rate-limit. |
| 4 | **Микросервисы (Catalog, Cart, Payment)** | Каждый отвечает за свой бизнес-домен; общаются между собой через REST/gRPC и Kafka. |
| 5 | **СУБД (PostgreSQL)** | Хранит каталоги, заказы, пользователей. Транзакционная нагрузка. |
| 6 | **Кеш (Redis)** | Хранит сессии, корзины, частые выборки. Снижает нагрузку на основную БД. |
| 7 | **Kafka** | Шина событий: «товар куплен» → бухгалтерия + рекомендации + аналитика. |
| 8 | **Elasticsearch** | Полнотекстовый поиск по каталогу. |

## 4. «Горячий» vs «холодный» резерв

| Критерий | Горячий резерв (Hot standby) | Холодный резерв (Cold standby) |
|----------|------------------------------|--------------------------------|
| **Готовность** | Резервная нода работает параллельно, готова принять нагрузку немедленно. | Резервное оборудование выключено или содержит только базовую установку, готовится при инциденте. |
| **Синхронизация данных** | Постоянная (репликация, кластерные технологии: streaming replication, синхронные/асинхронные слейвы). | Периодическая (бэкапы, дампы), либо нет вообще — данные восстанавливаются с резервной копии. |
| **Время переключения (RTO)** | Секунды — минуты (часто автоматический failover). | Часы — сутки (требует ручной развёртки и восстановления данных). |
| **Стоимость** | Высокая: 2× инфраструктура + постоянная работа. | Низкая: оборудование стоит выключенным/арендуется по запросу. |
| **Плюсы** | Минимальный простой, прозрачный для пользователя; подходит для бизнес-критичных систем. | Дёшево, просто реализовать, низкие операционные расходы. |
| **Минусы** | Дорого, требует мониторинга «split-brain», сложнее сопровождать. | Большой простой при инциденте; потенциальная потеря данных между бэкапами. |
| **Когда применять** | Платёжные системы, биржи, телеком, e-commerce под нагрузкой. | Внутренние утилиты, аналитика, dev/staging-окружения, архивные сервисы. |

Существует ещё промежуточный вариант — **«тёплый» резерв** (warm standby): нода запущена, но не обслуживает нагрузку, синхронизация данных периодическая, переключение требует минут.

## 5. Сценарий теста отказоустойчивости («горячий» резерв) для компонента PostgreSQL

**Цель теста:** убедиться, что при отказе master-узла БД сервис каталога Ozon продолжает обслуживать запросы пользователей с допустимой деградацией (RTO ≤ 30 секунд, потеря не более 1 транзакции).

### Предусловия

- Развёрнут кластер PostgreSQL: 1 master + 2 hot standby с потоковой репликацией (synchronous_commit = on).
- Перед master стоит pgbouncer/Patroni, обеспечивающий автоматический failover.
- Нагрузка имитируется через JMeter: 500 RPS на endpoint `GET /catalog/items` и 50 RPS на `POST /orders`.

### Шаги

1. Запустить нагрузочный профиль на 10 минут — собрать baseline (p50/p95/p99 latency, error rate).
2. На 5-й минуте выполнить `kill -9` процесса PostgreSQL на master-узле (имитация аппаратного сбоя).
3. Зафиксировать:
   - время до автоматического переключения на standby (по логам Patroni);
   - количество и тип ошибок, увиденных клиентом в этот период;
   - факт сохранения in-flight транзакций.
4. После promote одного из standby в новый master продолжить нагрузку ещё 5 минут.
5. Сравнить latency/error rate до и после failover.
6. Восстановить исходный master как standby и проверить, что репликация запустилась без потерь.

### Метрики для отслеживания

| Метрика | Цель |
|---------|------|
| **RTO (Recovery Time Objective)** — время от падения до восстановления обслуживания | ≤ 30 секунд |
| **RPO (Recovery Point Objective)** — допустимая потеря данных | 0 транзакций (синхронная репликация) |
| **Error rate во время failover** | ≤ 1% от общего числа запросов |
| **p95 latency после failover** | не более чем в 1,5× базового значения в течение первой минуты |
| **Replication lag (МБ/секунд)** между master и standby | ≤ 1 секунды на стабильной нагрузке |
| **Сообщения пользователю** | Корректная обработка ошибок: понятный текст «Попробуйте через минуту», без stacktrace |
| **Восстановление прежнего master** в роли standby | Без ручного вмешательства, lag → 0 за ≤ 5 минут |

### Что считаем провалом теста

- RTO > 60 секунд.
- Появление дублей заказов после failover.
- Утрата данных, подтверждённых клиенту 200 OK.
- Длительный «split-brain» (оба узла считают себя master).
