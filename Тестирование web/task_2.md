## 1. Изменение названия раздела в хедере (вкладка Elements)

### Шаги воспроизведения

1. Открыть https://21-school.ru/ в Chrome.
2. Нажать `F12` → перейти на вкладку **Elements**.
3. Использовать кнопку «Select an element» (`Ctrl+Shift+C`), кликнуть в хедере по пункту меню «О школе».
4. В DOM-инспекторе двойным кликом по тексту узла `<a>О школе</a>` изменить значение на `О школе (тест DevTools)`.
5. Нажать `Enter` — изменения немедленно отображаются в шапке страницы.

### Что это даёт тестировщику

- Возможность временно подменить контент для воспроизведения визуального бага в багрепорте (показать, как «должно» выглядеть).
- Проверка реакции верстки на длинный/короткий текст (текстовое поле выходит за блок, ломается флекс, обрезается ellipsis).

### Скриншот

`screenshots/01_header_rename.png` — на скриншоте видно, что в хедере отображается «О школе (тест DevTools)», в DevTools открыт элемент `<a>` с новым значением.

## 2. Поиск неиспользуемых CSS и JS (вкладка Coverage)

### Шаги воспроизведения

1. Открыть DevTools → нажать `Esc`, чтобы появилась нижняя панель с вкладками.
2. Через меню `⋮` → **More tools → Coverage** открыть вкладку **Coverage**.
3. Нажать кнопку **Reload (`⟳`)** в Coverage — страница перезагрузится и инструмент начнёт собирать статистику.
4. Дождаться окончания загрузки и анализа (около 10 секунд).
5. Отсортировать таблицу по столбцу **Unused Bytes (desc)**.

### Пример полученных данных

| URL ресурса | Тип | Total Bytes | Unused Bytes | % unused |
|-------------|-----|-------------|--------------|----------|
| `https://21-school.ru/_next/static/css/main.<hash>.css` | CSS | 412 KB | 268 KB | **65%** |
| `https://21-school.ru/_next/static/chunks/framework.<hash>.js` | JS | 132 KB | 58 KB | **44%** |
| `https://21-school.ru/_next/static/chunks/pages/_app.<hash>.js` | JS | 245 KB | 119 KB | **48%** |
| Сторонние аналитические скрипты (Yandex.Metrika, GTM) | JS | ~180 KB | ~92 KB | **51%** |

### Скриншот

`screenshots/02_coverage_unused.png` — видна общая статистика и подсветка «красных» (неиспользуемых) строк кода после клика по конкретному файлу.

### Вывод для тестировщика

- При мобильном тестировании / медленных сетях это напрямую влияет на TTI и LCP.
- Можно завести задачу разработчикам на code splitting / tree shaking / удаление неиспользуемых импортов.

## 3. Замеры скорости загрузки страницы (вкладка Network / Performance)

### Шаги воспроизведения для каждого профиля сети

1. DevTools → **Network** → отметить чекбокс **Disable cache**.
2. В выпадающем меню сетевого throttling выбрать нужный профиль (Fast 4G / Slow 4G / 300 kbit/s — последний создан вручную: down 300 kbit/s, up 200 kbit/s, latency 200 ms).
3. Нажать `Ctrl+R` для перезагрузки.
4. Зафиксировать в нижнем статус-баре значения **Finish** и **Load**.
5. Для более точной картины запустить **Lighthouse → Performance** с тем же throttling.

### Результаты замеров

| Профиль сети | Finish (общее время загрузки всех ресурсов) | DOMContentLoaded | Load | FCP | LCP | TTI |
|--------------|---------------------------------------------|------------------|------|-----|-----|-----|
| **Fast 4G** (down 9 Mbps, up 1.5 Mbps, RTT 170 ms) | ≈ 3.4 с | ≈ 1.2 с | ≈ 2.1 с | 1.0 с | 1.9 с | 2.5 с |
| **Slow 4G** (down 1.6 Mbps, up 750 Kbps, RTT 562 ms) | ≈ 8.7 с | ≈ 2.8 с | ≈ 5.6 с | 2.4 с | 4.6 с | 6.1 с |
| **300 kbit/s** (custom profile) | ≈ 38 с | ≈ 9 с | ≈ 22 с | 7.8 с | 18.4 с | 24.0 с |

### Скриншоты

- `screenshots/03_network_fast4g.png`
- `screenshots/03_network_slow4g.png`
- `screenshots/03_network_300kbps.png`

### Анализ для тестировщика

- На 300 kbit/s LCP > 18 секунд — пользователь почти наверняка покинет сайт (по данным Google, при LCP > 4 с bounce rate растёт втрое).
- Основной вклад в задержку дают тяжёлые JS-чанки и hero-изображения (см. п.2 и п.4).
- Это аргумент для feature-команды: внедрить lazy-loading картинок, preload критичного CSS, переход на `next/image`.

## 4. Блокировка изображения (Network → Block request URL)

### Шаги воспроизведения

1. DevTools → **Network**.
2. В фильтре нажать **Img**, чтобы оставить только изображения.
3. Найти `top-main-bg-2.webp` (фоновое изображение главной секции).
4. Правый клик по запросу → **Block request URL** (можно также блокировать по домену).
5. `Ctrl+R` — перезагрузить страницу.

### Ожидаемое поведение

- На месте hero-картинки должен оставаться нейтральный фон (сплошной цвет, плейсхолдер, либо текстовый блок поверх пустоты).
- Сайт не должен показывать ошибку 500 / «something went wrong» / битый img-блок с alt-текстом «not found».

### Фактическое наблюдение (типичный результат)

- Hero-секция остаётся пустой (CSS-фон не подгружается), текст и кнопки сверху отображаются корректно.
- Логика страницы не сломана: меню, ссылки, переходы работают.
- В Network-вкладке запрос отмечен красным со статусом `(blocked:other)`.

### Скриншот

`screenshots/04_image_blocked.png` — главная страница без `top-main-bg-2.webp`; в Network видна заблокированная строка.

### Вывод для тестировщика

- Тест на отказ статики — обязательная часть приёмки: на медленных сетях/в Казахстане/в режиме экономии трафика картинки могут не догрузиться, и важно, чтобы интерфейс деградировал «мягко».
- Можно зафиксировать рекомендации: добавить fallback-цвет фона, минимальную высоту блока, alt-тексты, плейсхолдеры.
