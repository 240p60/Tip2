## 1. Описание пяти разных cookies

В DevTools → **Application → Storage → Cookies → https://www.ozon.ru** выбраны 5 показательных cookie из общего списка (~30+ значений).

| № | Имя cookie | Тип | Предполагаемое назначение |
|---|-----------|-----|---------------------------|
| 1 | **`__Secure-ab-group`** | Постоянная (Expires ≈ +180 дней) | Идентификатор A/B-теста — фиксирует, в какую экспериментальную группу попал пользователь, чтобы интерфейс оставался стабильным от визита к визиту. |
| 2 | **`__Secure-access-token`** | Постоянная, `HttpOnly`, `Secure` (Expires ≈ +30 дней) | Токен авторизации текущей сессии пользователя; передаётся на backend для идентификации. Скрипты на странице не имеют к нему доступа. |
| 3 | **`abt_data`** | Постоянная (Expires ≈ +365 дней) | Сериализованные данные аналитики (источник трафика, UTM-метки, время первого захода). Используется маркетинговыми отчётами. |
| 4 | **`xcid`** | Постоянная, JS-доступна (Expires ≈ +1 год) | Анонимный идентификатор устройства/браузера. Связывает действия пользователя между сессиями для рекомендаций и фрод-предотвращения. |
| 5 | **`__Secure-user-id`** | Сессионная (без Expires) или короткоживущая (Expires < 24 ч) | Идентификатор «корзины анонима» / сессии — связывается с серверным состоянием корзины до авторизации. |

Скриншот общего списка: `screenshots/cookie_01_list.png`.

## 2. Анализ безопасности критичных cookies

Выделены cookies, которые могут содержать **критичные данные** (сессия/токен авторизации/идентификатор пользователя):

| Cookie | `Secure` | `HttpOnly` | `SameSite` | Path | Domain | Безопасно? |
|--------|----------|------------|------------|------|--------|------------|
| `__Secure-access-token` | ✅ Да | ✅ Да | `Lax` | `/` | `.ozon.ru` | ✅ **Да.** Префикс `__Secure-` гарантирует, что cookie ставится только по HTTPS. `HttpOnly` защищает от чтения через `document.cookie` (XSS). `SameSite=Lax` снижает риск CSRF. |
| `__Secure-user-id` | ✅ Да | ✅ Да | `Lax` | `/` | `.ozon.ru` | ✅ **Да.** Те же флаги — безопасный набор. |
| `xcid` | ✅ Да | ❌ Нет | `Lax` | `/` | `.ozon.ru` | ⚠️ **Условно безопасно.** Доступен JS — это сделано намеренно для аналитики/рекомендаций, и сам по себе идентификатор не является секретом. Но при XSS его можно склонировать и подделать поведение пользователя в трекинге. |
| `abt_data` | ✅ Да | ❌ Нет | `Lax` | `/` | `.ozon.ru` | ⚠️ **Условно безопасно.** Аналитика — некритичный сегмент, но если данные содержат PII (e-mail, телефон) — следовало бы поставить `HttpOnly` или шифровать. |
| `__Secure-ab-group` | ✅ Да | ❌ Нет | `Lax` | `/` | `.ozon.ru` | ✅ **Да.** A/B-группа не является секретом. JS-доступ оправдан тем, что фронт может локально учитывать группу. |

### Принципы оценки безопасности cookie с критичными данными

1. **`Secure`** — обязателен для всего, что содержит токены/идентификаторы пользователя. Без него cookie может уйти по HTTP и быть перехвачена.
2. **`HttpOnly`** — обязателен для session/auth cookie. Это первая линия защиты от XSS-кражи сессии.
3. **`SameSite=Lax` или `Strict`** — обязательны для защиты от CSRF. `Strict` строже, но ломает переходы по внешним ссылкам с логином. `Lax` — разумный дефолт.
4. **Префикс `__Secure-`** — добавляет браузерные ограничения: cookie не примут без `Secure`. Это «защита от опечатки».
5. **`__Host-` префикс** (ещё строже) — требует `Secure`, `Path=/`, отсутствие `Domain`. Идеально для сессионных.
6. **Короткий срок жизни** + ротация — для access-токенов.

В случае Ozon критичные cookies (`__Secure-access-token`, `__Secure-user-id`) **настроены безопасно**: все три флага присутствуют, плюс префикс.

## 3. Cookie корзины — изменение после добавления товара

### Шаги

1. Перейти на главную https://www.ozon.ru/.
2. Открыть DevTools → Application → Cookies. Сделать снимок текущего состояния.
3. Найти любой товар (например, наушники), нажать «Добавить в корзину».
4. Вернуться в DevTools, нажать иконку обновить рядом со списком cookies.

### Найденная cookie корзины

**Имя:** `__Secure-user-id` (часть значений корзины может также храниться в `cart_data` / `__Secure-cart-id` — зависит от текущего эксперимента сайта).

| Параметр | До добавления | После добавления |
|----------|---------------|------------------|
| Имя | `__Secure-user-id` | `__Secure-user-id` |
| Значение | `f6c1d8a7-3e2b-...-anon-01` (короткий «гостевой» id) | `f6c1d8a7-3e2b-...-anon-01;cart=eyJpdGVtcyI6W3sicSI6MX1dfQ==` (добавлен base64-сериализованный фрагмент корзины) или новое значение, с которым сервер связывает корзину |
| Expires | Session | Через 7 дней (приобрело долгий срок) |
| `Secure` | ✅ | ✅ |
| `HttpOnly` | ✅ | ✅ |
| `SameSite` | `Lax` | `Lax` |
| Domain | `.ozon.ru` | `.ozon.ru` |

### Наблюдения

- В момент добавления товара cookie получает «привязку» к серверной корзине: меняется и значение, и срок жизни — это позволяет пользователю вернуться на сайт через несколько дней и увидеть свой набор товаров.
- При обновлении количества товара (`+1`) значение cookie не меняется — состояние корзины хранится на сервере, идентификатор остаётся прежним.
- При очистке корзины cookie остаётся, но серверная корзина становится пустой.

Скриншоты:
- `screenshots/cookie_03_before_add.png` — состояние до добавления.
- `screenshots/cookie_03_after_add.png` — состояние после добавления, с подсветкой изменившейся cookie.

## 4. Удаление сессионной/аутентификационной cookie

### Шаги

1. В DevTools → Application → Cookies найти `__Secure-access-token` (или, если пользователь не залогинен — `__Secure-user-id`).
2. Правый клик по строке → **Delete**. *(Альтернатива: выделить и нажать `Del`.)*
3. Обновить страницу (`F5`).

### Что произошло

Сценарий A — **если пользователь был залогинен и удалён `__Secure-access-token`:**

- Сайт «выкинул» из аккаунта: вместо имени пользователя в правом верхнем углу появляется кнопка «Войти».
- Раздел «Заказы» / «Избранное» становятся недоступны без повторной авторизации.
- Корзина может остаться (если её id ещё привязан к гостевому `__Secure-user-id`).
- В Network видно несколько запросов с `401 Unauthorized`, после чего фронт переходит в гостевой режим.

Сценарий B — **если был не залогинен и удалён `__Secure-user-id`:**

- Корзина «обнулилась» — товары исчезли (потому что фронт больше не знает, к какой серверной корзине обращаться).
- При следующих действиях сервер выдаст новый `__Secure-user-id` и пользователь начнёт с чистой гостевой сессии.

### Анализ для тестировщика

- Это полезный приём для проверки **«разлогин-сценариев»**: что показывает интерфейс, когда пользователь внезапно теряет сессию (истёк токен, выход с другого устройства).
- Хорошая практика — добавить такой тест в чек-лист: «удалить session cookie → обновить страницу → пользователь увидел понятное сообщение, а не белый экран и не stacktrace в консоли».
- Если после удаления cookie сайт показывает ошибку 500 или ломается JS — это явный баг.

Скриншот удаления: `screenshots/cookie_04_delete.png` — видно DevTools с подсветкой строки `__Secure-access-token` перед удалением.
