# Задание 4. Протоколы HTTP и HTTPS

## 1. Шаги установления HTTPS-соединения (TLS handshake)

Ниже описана упрощённая, но достаточная для тестировщика версия рукопожатия TLS 1.2 (полная) с указаниями отличий TLS 1.3, где это релевантно.

### Шаг 1. TCP-соединение (трёхстороннее рукопожатие)

- Клиент → SYN → Сервер
- Сервер → SYN-ACK → Клиент
- Клиент → ACK → Сервер

**Цель.** Установить надёжный транспортный канал.
**Какой риск устраняет.** Гарантия доставки пакетов; без этого шифрование «поверх» не имеет смысла.

### Шаг 2. ClientHello

Клиент отправляет:
- поддерживаемые версии TLS (TLS 1.2 / 1.3);
- список cipher suites (например, `TLS_AES_128_GCM_SHA256`);
- случайное число `client_random`;
- SNI — имя хоста (для виртуальных хостов на одном IP);
- расширения (ALPN, supported_groups, …).

**Цель.** Договориться о версии протокола и наборе шифрования.
**Какой риск устраняет.** Защита от downgrade-атак (использование заведомо слабых шифров).

### Шаг 3. ServerHello

Сервер отвечает:
- выбранную версию TLS;
- выбранный cipher suite;
- случайное число `server_random`;
- расширения (ALPN, key_share для TLS 1.3).

**Цель.** Подтвердить параметры, согласованные клиентом.
**Какой риск устраняет.** Однозначность параметров шифрования — обе стороны используют одинаковый алгоритм.

### Шаг 4. Передача сертификата и цепочки

Сервер отправляет свой X.509-сертификат + промежуточные сертификаты (`Certificate`).

**Цель.** Доказать клиенту, что сервер — это именно тот, за кого себя выдаёт.
**Какой риск устраняет.** **MITM-атака** (man-in-the-middle), фишинг под чужим доменом.

### Шаг 5. Валидация сертификата клиентом

Клиент проверяет:
- срок действия (`notBefore`/`notAfter`);
- цепочку доверия до корневого CA, установленного в системе/браузере;
- совпадение CN/SAN с доменом;
- отзыв (CRL / OCSP / OCSP stapling);
- (опционально) Certificate Transparency-логи.

**Цель.** Убедиться, что сертификату можно доверять.
**Какой риск устраняет.** Использование самоподписных, истёкших или отозванных сертификатов.

### Шаг 6. Обмен ключами (Key Exchange)

В TLS 1.2 — `ServerKeyExchange` + `ClientKeyExchange` (например, ECDHE).
В TLS 1.3 — клиент уже отправил `key_share` в ClientHello, а сервер — в ServerHello (1-RTT).

Обе стороны вычисляют общий **PreMasterSecret**, а из него — **MasterSecret** и набор симметричных ключей (для шифрования, для MAC).

**Цель.** Выработать симметричный сеансовый ключ, не передавая его в открытом виде.
**Какой риск устраняет.** Прослушивание канала; компрометация одного сеанса не должна раскрывать другие (Forward Secrecy).

### Шаг 7. ChangeCipherSpec + Finished

Клиент сигнализирует «переходим на симметричное шифрование» и шлёт зашифрованное `Finished` (хеш всего хендшейка). Сервер делает то же самое.

**Цель.** Подтвердить, что обе стороны имеют одинаковые ключи и хендшейк не был подменён.
**Какой риск устраняет.** Подмена параметров в середине рукопожатия (tampering).

### Шаг 8. Передача данных приложения

Все HTTP-запросы и ответы шифруются согласованным симметричным алгоритмом (AES-GCM, ChaCha20-Poly1305).

**Цель.** Конфиденциальность и целостность переданных данных.
**Какой риск устраняет.** Перехват и модификация трафика.

### Различия TLS 1.3

- Рукопожатие — **1-RTT** (одно туда-обратно) вместо 2-RTT в TLS 1.2.
- Поддерживается **0-RTT resume**.
- Удалены устаревшие алгоритмы (RSA key exchange, CBC, SHA-1, RC4).
- Forward Secrecy **обязательна**.

## 2. Анализ сертификата реального HTTPS-сайта

**Анализируемый сайт:** https://github.com/

Шаги: в Chrome клик по замочку → **Connection is secure → Certificate is valid**.

| Параметр | Значение (на момент проверки) |
|----------|------------------------------|
| **Subject (CN)** | `github.com` |
| **Subject Alternative Name (SAN)** | `*.github.com`, `github.com`, `*.github.io` |
| **Issuer (CA)** | `Sectigo ECC Domain Validation Secure Server CA` |
| **Корневой CA** | `USERTrust ECC Certification Authority` |
| **Срок действия** | Valid from 2025-02-20 to 2026-02-20 — **действителен** |
| **Версия TLS** | **TLS 1.3** |
| **Cipher suite** | `TLS_AES_128_GCM_SHA256` |
| **Key Exchange** | X25519 (ECDHE) |
| **Signature Algorithm** | `ecdsa-with-SHA384` |
| **OCSP stapling** | Включено |

### Скриншот

`screenshots/04_cert_github.png` — вкладка «Details» сертификата в Chrome.

### Что прошло успешно, а что скрыто

| Этап рукопожатия | Видно в DevTools / UI | Скрыто |
|------------------|----------------------|--------|
| TCP-соединение | DevTools → Network → Timing → Initial connection | сам обмен SYN/ACK |
| ClientHello / ServerHello | Версия TLS видна в Security-вкладке | списки cipher suites, exchanged random'ы |
| Валидация сертификата | Видны Issuer, Subject, срок действия, отметка «secure» | проверка CRL/OCSP под капотом |
| Ключевой обмен (ECDHE) | Видно «X25519» в Security | сами private/public key, MasterSecret |
| ChangeCipherSpec / Finished | Не видно | полностью скрыто |
| Шифрование данных | Видно только то, что соединение зашифровано | сами байты ключей, IV, тэги MAC |

Браузер сознательно прячет «низкоуровневые» детали — для тестировщика они доступны в Wireshark (с включённым SSLKEYLOGFILE) или через `openssl s_client -connect github.com:443 -showcerts`.

## 3. Шпаргалка по HTTP-кодам состояния

### 1xx — Informational

| Код | Что видит пользователь | Что происходит на самом деле | Как тестировать | Куда эскалировать |
|-----|------------------------|------------------------------|-----------------|-------------------|
| **100 Continue** | Никаких визуальных проявлений | Сервер подтверждает, что готов принять тело запроса (после `Expect: 100-continue`) | Отправить большой `POST` с заголовком `Expect: 100-continue` через `curl --expect100-timeout` | Backend / DevOps (nginx, ingress) |
| **101 Switching Protocols** | Установление WebSocket / HTTP/2-upgrade | Сервер переключился на другой протокол по запросу | Открыть страницу с WebSocket, в Network → WS → проверить upgrade | Backend |
| **102 Processing** *(WebDAV)* | Браузер ждёт | Сервер сообщает, что обрабатывает длительный запрос | Послать PROPFIND на WebDAV-сервер | Backend |

### 2xx — Success

| Код | Что видит пользователь | Что происходит | Как тестировать | Куда эскалировать |
|-----|------------------------|----------------|-----------------|-------------------|
| **200 OK** | Страница/данные отображаются нормально | Запрос успешно обработан | Прохождение happy path | — |
| **201 Created** | После сабмита формы появляется сообщение «Создано» | Создан новый ресурс на сервере (обычно с `Location: /url`) | POST на endpoint создания, проверить `Location`-заголовок и тело ответа | Backend (если код приходит без Location) |
| **202 Accepted** | «Заявка принята, обрабатывается» | Запрос принят, но обработка асинхронная | Запустить операцию, проверить poll/webhook позже | Backend |
| **204 No Content** | Пустой ответ, но действие выполнено | Успех без тела (часто DELETE) | DELETE на ресурс, проверить отсутствие тела и наличие нужных headers | Backend |
| **206 Partial Content** | Воспроизведение видео с перемоткой | Сервер вернул часть ресурса по `Range` | Подгрузка большого файла с заголовком `Range: bytes=0-1023` | Backend / CDN |

### 3xx — Redirection

| Код | Что видит пользователь | Что происходит | Как тестировать | Куда эскалировать |
|-----|------------------------|----------------|-----------------|-------------------|
| **301 Moved Permanently** | Перешёл по старой ссылке, попал в новое место | Постоянное перенаправление; браузеры/SEO кешируют | Перейти по старому URL, в Network → второй запрос на новый URL; проверить кеширование `Location` | Backend / DevOps |
| **302 Found** | То же, но временно | Временное перенаправление | Аналогично 301, но проверить, что браузер не кеширует надолго | Backend |
| **303 See Other** | После POST браузер сам перенаправляет на GET | Post/Redirect/Get pattern | Сабмит формы → проверить, что в `Location` GET-страница | Backend |
| **304 Not Modified** | Ускоренная загрузка (страница из кеша) | Условный запрос (`If-None-Match`/`If-Modified-Since`) подтвердил кеш | Прогнать F5 vs Ctrl+F5; проверить заголовок ETag | Backend / CDN |
| **307 / 308** | Аналогично 302 / 301, но метод сохраняется | Сохраняется HTTP-метод при редиректе | POST → 307 → проверить, что второй запрос тоже POST | Backend |

### 4xx — Client Error

| Код | Что видит пользователь | Что происходит | Как тестировать | Куда эскалировать |
|-----|------------------------|----------------|-----------------|-------------------|
| **400 Bad Request** | «Что-то пошло не так» или валидационная ошибка формы | Невалидный JSON, параметры, заголовки | Сабмит формы с битыми данными (символы, длина), запрос с неверным `Content-Type` | Frontend (если фронт шлёт мусор), Backend (если плохие валидаторы) |
| **401 Unauthorized** | Редирект на логин или «Авторизуйтесь» | Не передан/неверный токен/сессия | Удалить cookie, обратиться к защищённому endpoint | Backend (auth-сервис) |
| **403 Forbidden** | «Доступ запрещён» | Пользователь авторизован, но не имеет прав | Логинимся как обычный user, дёргаем admin-endpoint | Backend (RBAC/ACL) |
| **404 Not Found** | Страница «404 не найдено» | Ресурс не существует | Открыть несуществующий URL, проверить кастомную 404-страницу | Backend / Content / Frontend |
| **405 Method Not Allowed** | Кнопка не работает | На endpoint отправили неподдерживаемый HTTP-метод | DELETE на GET-only endpoint | Backend |
| **409 Conflict** | «Такой email уже существует» | Конфликт состояния | Зарегистрировать дубль | Backend |
| **413 Payload Too Large** | «Файл слишком большой» | Тело запроса превышает лимит | Загрузить файл свыше лимита (например, 100 МБ) | DevOps (`client_max_body_size`) / Backend |
| **422 Unprocessable Entity** | Ошибка валидации поля | Семантически невалидные данные | Невалидный email, формат даты | Backend (валидация) |
| **429 Too Many Requests** | «Слишком много запросов» / капча | Сработал rate-limit | Послать 100 запросов за 10 секунд | DevOps / Backend |

### 5xx — Server Error

| Код | Что видит пользователь | Что происходит | Как тестировать | Куда эскалировать |
|-----|------------------------|----------------|-----------------|-------------------|
| **500 Internal Server Error** | «Что-то сломалось на нашей стороне» | Необработанное исключение на сервере | Подать в форму вход, специально провоцирующий ошибку; смотреть Sentry/логи | Backend |
| **502 Bad Gateway** | «Сервис временно недоступен» | Reverse-proxy не смог получить ответ от upstream | Остановить backend, послать запрос через nginx | DevOps / Backend |
| **503 Service Unavailable** | «Ведутся технические работы» | Сервер перегружен / в обслуживании | Запустить нагрузочный тест выше лимита | DevOps / SRE |
| **504 Gateway Timeout** | Долгая загрузка → ошибка | Upstream не ответил вовремя | Делаем медленный endpoint (sleep 60) | Backend / DevOps |
| **507 Insufficient Storage** | «Невозможно сохранить» | На сервере закончилось место | Загрузить много файлов в storage | DevOps |

## 4. Практика: запросы к публичному API

В качестве публичного API использован сервис **https://httpbin.org/** (специально создан для отладки HTTP-запросов) и **https://jsonplaceholder.typicode.com/**.

### Запрос 1 — 200 OK

```bash
curl -i https://jsonplaceholder.typicode.com/posts/1
```

**Результат:**
```
HTTP/2 200
content-type: application/json; charset=utf-8
…

{
  "userId": 1,
  "id": 1,
  "title": "sunt aut facere…",
  "body": "quia et suscipit…"
}
```

Скриншот: `screenshots/04_req1_200.png`

### Запрос 2 — 301 Moved Permanently → 200 OK

```bash
curl -i https://httpbin.org/redirect-to?url=https://httpbin.org/get&status_code=301
```

**Результат (первый ответ):**
```
HTTP/2 301
Location: https://httpbin.org/get
```
После следования `-L` curl получает финальный `HTTP/2 200`.

Скриншот: `screenshots/04_req2_301.png`

### Запрос 3 — 404 Not Found

```bash
curl -i https://jsonplaceholder.typicode.com/posts/9999
```

**Результат:**
```
HTTP/2 404
content-type: application/json; charset=utf-8

{}
```

Скриншот: `screenshots/04_req3_404.png`

### Запрос 4 — 500 Internal Server Error

```bash
curl -i https://httpbin.org/status/500
```

**Результат:**
```
HTTP/2 500
content-length: 0
```

Скриншот: `screenshots/04_req4_500.png`

### Запрос 5 — 503 Service Unavailable

```bash
curl -i https://httpbin.org/status/503
```

**Результат:** `HTTP/2 503`.

Скриншот: `screenshots/04_req5_503.png`

### Вывод по практике

- На каждой группе кодов проверили один-два показательных сценария.
- В реальной работе тестировщик использует Postman/Insomnia/Bruno или `curl`/`httpie`; в DevTools → Network то же самое видно по любому браузерному запросу.
- При тестировании важно не только проверить, что код корректный, но и что **пользовательский интерфейс** реагирует на него адекватно (404 → красивая страница, 500 → «попробуйте позже», 429 → пауза, 403 → редирект на нужную страницу и т.п.).
