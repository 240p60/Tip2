## 3. Почему это баг, а не фича / ожидаемое поведение

1. **Противоречит цели формы.** Поле явно помечено как обязательное (`*` и валидационное сообщение «Last Name is required»). Пользователь физически не может выполнить требование, которое продукт сам же предъявляет, — это логическое противоречие.
2. **Несогласованность с другими полями.** Поля «First Name» и «Postal Code» работают корректно — значит, проблема не в общей блокировке формы, а в конкретном инпуте. Если бы это было задумано (например, демо-режим), отключение должно было быть консистентным.
3. **Несогласованность с другими пользователями.** Для `standard_user` поле «Last Name» работает штатно. Различие именно для `problem_user` — это и есть смысл существования этого аккаунта на сайте: демонстрировать дефекты QA-инженерам, чтобы они их находили и репортили.
4. **Блокирует ключевой пользовательский сценарий.** Невозможно завершить покупку, что нарушает основной бизнес-флоу e-commerce (логин → каталог → корзина → чекаут → оплата). Любая фича, ломающая основной флоу без явной причины, — это дефект.
5. **Нет упоминания такого поведения в документации/UI.** Никаких подсказок «поле временно отключено» нет — пользователь воспринимает поле как активное.

Вывод: поведение не может быть классифицировано как фича или ожидаемое — это явный функциональный дефект.

## 4. Гипотеза о причинах и точках исправления

### Технические гипотезы

1. **Программная блокировка ввода на уровне обработчика события.** Возможно, для `problem_user` к input навешан обработчик `onKeyDown` / `onChange`, который вызывает `event.preventDefault()` либо обнуляет значение в state. Точка исправления — компонент формы чекаута (`CheckoutInfo` / аналог), условный код типа `if (user === 'problem_user') ...`.
2. **Атрибут `readonly` или `disabled`, выставляемый условно.** Поле может получать атрибут `readonly` через JS при загрузке страницы для конкретного пользователя. Точка проверки — DOM-инспектор, точка исправления — рендер инпута.
3. **Подмена `name`/`id` поля.** На форме два инпута с одинаковым `name` (например, у First Name и Last Name `name="firstName"`), и значение перезаписывается. Точка исправления — JSX/HTML формы.
4. **Регрессия на стороне валидации.** Валидатор может «съедать» значение, если оно не проходит regex (но это маловероятно при пустом инпуте).

### Как проверить (для разработчика)

- Открыть DevTools → Elements → найти `input` с `placeholder="Last Name"` и проверить наличие атрибутов `readonly`, `disabled`, и одинаковость `id/name`.
- Открыть DevTools → Sources → поставить breakpoint на изменения state формы и попробовать ввести символ.
- Сравнить рендер формы для `standard_user` и `problem_user` (diff DOM).

### Точки исправления

- Файл/компонент, отвечающий за рендер `Checkout: Your Information`.
- Логика «искусственных дефектов» для `problem_user` — если такая существует, скорее всего, всё локализовано в одном месте.

## 5. Влияние на пользователя и бизнес-процессы

### Риски и неудобства для пользователя

- **Невозможность завершить покупку.** Финальная и самая критичная стадия воронки полностью заблокирована — пользователь застревает на чекауте.
- **Потеря доверия.** Пользователь видит ошибку «Last Name is required», но не понимает, что он его «вводит» — складывается ощущение неработающего сайта или собственной ошибки.
- **Зря потраченное время.** Пользователь прошёл всю воронку (логин, выбор товара, корзина) — и упёрся в стену на финальном шаге. Это самый болезненный сценарий с точки зрения UX.
- **Возможные доступные обходы — отсутствуют.** Пользователь не может «пропустить» поле, не может ввести значение через автозаполнение (если это всё блокируется на уровне state).

### Риски для бизнеса

- **Прямая потеря выручки.** Конверсия в покупку = 0 для всех пользователей, попадающих в этот сценарий. Если это реальный e-commerce — это прямые финансовые потери, пропорциональные доле трафика.
- **Рост cart abandonment rate.** Метрика «брошенных корзин» вырастет, что испортит маркетинговую аналитику и приведёт к ложным гипотезам («у нас плохой UX корзины»).
- **Удар по репутации.** Пользователи будут жаловаться в соцсетях и поддержку («не могу оформить заказ»), что ухудшает NPS и формирует негативные отзывы в интернете.
- **Нагрузка на support.** Каждый такой случай — обращение в поддержку с непонятным симптомом, которое тяжело воспроизвести (зависит от учётной записи / сегмента пользователей).
- **Юридические/регуляторные риски.** Для регулируемых индустрий (фарма, финтех) недоступность сервиса для части пользователей может квалифицироваться как нарушение SLA / договора.
- **Затраты на расследование.** Если баг попадёт в прод, на разбор уйдут часы инженеров (отдел QA + dev + DevOps + аналитика), пока проблему не локализуют — особенно если она «плавающая» по сегменту пользователей.

### Приоритезация для бизнеса

Учитывая, что дефект:
- блокирует сквозной user journey (purchase),
- воспроизводится со 100% вероятностью для соответствующего сегмента,
- виден на финальном этапе воронки (наиболее дорогостоящие пользователи),

→ **исправление должно идти в hotfix-релизе** до любых других фич.
