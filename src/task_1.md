# Задание 1. Виды дефектов

Ниже представлены 5 известных в истории IT багов с анализом по основным критериям.

| Вид дефекта | Почему не был обнаружен при тестировании? | Какие виды тестирования могли бы его найти? | Экономические/репутационные последствия |
|-------------|-------------------------------------------|---------------------------------------------|-----------------------------------------|
| **Ariane 5 (1996)** — функциональный/арифметический дефект: переполнение при преобразовании 64-битного числа с плавающей точкой в 16-битное целое в инерциальной навигационной системе, унаследованной из Ariane 4. | Тестирование не покрыло сценарии полёта Ariane 5: не учли, что горизонтальная скорость новой ракеты превышает диапазон, заложенный для Ariane 4. Модуль считался «проверенным», поэтому повторно не верифицировался в новых условиях. Фактического end-to-end запуска перед коммерческим стартом не было. | Интеграционное тестирование с реальной телеметрией Ariane 5; нагрузочное/граничное тестирование (boundary value analysis) с диапазонами скоростей; тестирование на основе реальных полётных данных (data-driven); регрессионное тестирование при переиспользовании legacy-кода. | Прямые потери ~370 млн долларов (ракета + спутники Cluster). Срыв научной программы ESA. Серьёзный удар по репутации Arianespace, задержка коммерческих запусков. Кейс стал классическим примером в учебниках по тестированию ПО. |
| **Therac-25 (1985–1987)** — критический дефект безопасности: race condition в самодельной многозадачной ОС медицинского ускорителя, позволяющий выдать пациенту дозу радиации в ~100 раз больше нормы. | Производитель (AECL) полностью полагался на программную блокировку, удалив аппаратные предохранители, которые были в моделях Therac-6 и Therac-20. Тестирование не моделировало быстрый ввод оператора (быстрая правка параметров за <8 секунд), который и провоцировал состояние гонки. Не проводился независимый аудит safety-critical кода. | Тестирование многопоточности и race conditions; стресс-тестирование UI на быстрый ввод; формальная верификация safety-critical частей; тестирование с реальными операторами (UAT); анализ FMEA (Failure Mode and Effects Analysis); fuzz-тестирование последовательности команд. | Минимум 6 пациентов получили смертельные/тяжёлые передозировки, 3 погибших. Серьёзные иски, отзыв всех аппаратов. Поворотный кейс: после него ужесточены стандарты разработки медицинского ПО (IEC 62304, FDA). Репутация AECL уничтожена. |
| **Knight Capital (2012)** — конфигурационный/релизный дефект: при деплое нового кода на 8 серверов один сервер не обновили, и старый «мёртвый» флаг переиспользовал переменную, что привело к неконтролируемой отправке рыночных ордеров. | Не было автоматического чек-листа развёртывания на все сервера; отсутствовал смоук-тест после релиза; флаг переиспользовали вместо того, чтобы создать новый (нарушение принципа feature toggle); отсутствовал kill-switch на бизнес-уровне. | Деплой-тестирование (deployment verification); канареечный релиз/blue-green deployment; smoke-тестирование в продакшене; тестирование процедур отката (rollback); chaos-тестирование (что произойдёт при рассогласованной версии серверов); мониторинг аномалий в реальном времени. | Убыток ~440 млн долларов за 45 минут. Падение акций компании на ~75%. Knight Capital фактически прекратил независимое существование — был поглощён Getco. Усиление требований SEC к pre-trade risk controls. |
| **Mars Climate Orbiter (1999)** — интеграционный дефект: один модуль использовал английские (фунт-сила-секунды), другой — метрические единицы (ньютон-секунды) для расчёта импульса двигателей. | Команды Lockheed Martin и JPL разрабатывали подсистемы независимо. Контракт интерфейса был, но не было автотестов на проверку единиц измерения. Тестирование подсистем велось изолированно, end-to-end-симуляция всей траектории не выявила расхождения, потому что калибровка была «настроена» под наблюдаемые данные. | Интеграционное тестирование между подсистемами с проверкой единиц измерения; тестирование контрактов API (contract testing); сквозное моделирование траектории (simulation testing); статический анализ типов с физическими единицами; code review на междисциплинарных границах. | Потеря аппарата стоимостью 327,6 млн долларов. Срыв марсианской миссии. Имиджевые потери NASA, пересмотр процессов кросс-командной разработки. Стандартный кейс на лекциях по интеграционному тестированию. |
| **Pentium FDIV bug (1994)** — аппаратно-вычислительный дефект: ошибка в таблице поиска (lookup table) блока деления процессора Intel Pentium, приводящая к неточным результатам деления некоторых чисел с плавающей точкой. | В ROM-таблице деления было 5 пропущенных ячеек из 1066. Внутреннее тестирование Intel ограниченным числом тест-векторов не задело эти конкретные комбинации. Дефект был статистически редким (~1 раз на 9 млрд операций для случайных данных), что усложняло обнаружение. Intel изначально решил, что проблема несущественна для большинства пользователей. | Полное (exhaustive) тестирование критичных арифметических операций; формальная верификация микроархитектуры; property-based testing для математических свойств (например, a/b * b ≈ a); регрессионное тестирование на эталонных вычислениях; внешний независимый аудит. | Прямые расходы Intel на отзыв и замену процессоров — ~475 млн долларов. Резкое падение доверия к бренду. Изменение PR-стратегии Intel: после кейса — открытое признание дефектов и оперативные программы замены. Стандартный пример в курсах по верификации hardware. |

## Краткие выводы

- Большинство «громких» багов — это не банальные опечатки, а **дефекты на стыке дисциплин** (код ↔ железо, команда ↔ команда, тест-окружение ↔ прод).
- Объединяющий фактор — **переоценка надёжности** уже работавших ранее компонентов (Ariane 5 переиспользовал код Ariane 4, Therac-25 — софт Therac-20).
- Сильнее всего «спасают» от подобных дефектов **интеграционное и сценарное тестирование на реальных данных**, а не модульные тесты.
