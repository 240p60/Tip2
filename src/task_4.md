# Задание 4. Жизненный цикл баг-репорта

## 1. Таблица жизненного цикла баг-репорта

Ниже представлены типовые этапы/статусы, через которые может проходить баг-репорт от создания до закрытия. Конкретный набор статусов зависит от трекера (Jira, YouTrack, Bugzilla, Redmine), но общая логика одинакова.

| Этап / статус жизненного цикла баг-репорта | Что происходит | Кто и на каком этапе может менять статус баг-репорта | Пример |
| :--- | :--- | :--- | :--- |
| **New** | Дефект создан, но в работу ещё не отправлен (возможно, тестировщик уточняет какие-либо детали для данного дефекта). | Статус выставляется автоматически при создании баг-репорта в трекере. | Тестировщик создал баг про ошибку «Last Name» на чекауте; пока дополняет шаги и прикрепляет логи. |
| **Assigned / Open** | Баг-репорт распределён на конкретного исполнителя (разработчика или команду) для анализа и фикса. | Меняет лид/менеджер/тимлид (или автоматически по правилам трекера). | Тимлид фронтенда назначает баг на разработчика, отвечающего за модуль Checkout. |
| **In Progress** | Разработчик принял баг в работу и приступил к его исправлению. | Разработчик, на которого назначен баг. | Разработчик дебажит компонент CheckoutInfo, ищет причину блокировки ввода. |
| **Need Info / Need More Info** | Разработчику недостаточно информации для воспроизведения или фикса; запрос дополнительных данных у тестировщика. | Разработчик переводит в этот статус; обратно — тестировщик после ответа. | «Не воспроизводится у меня. Приложите HAR и версию браузера». |
| **Resolved / Fixed** | Дефект исправлен в коде, изменения смержены в нужную ветку. Ждёт проверки тестировщиком. | Разработчик. | Разработчик закоммитил фикс в ветку `develop`, поставил статус Fixed. |
| **Ready for Test / Ready for QA** | Сборка с фиксом доступна на тестовом стенде, баг готов к ретесту. | Разработчик / релиз-инженер / автоматически после деплоя. | Билд `1.4.7-rc2` выкачен на staging, баг попал в список ретеста. |
| **In Verification / Verifying** | Тестировщик прогоняет ретест по шагам воспроизведения и проверяет смежные сценарии. | Тестировщик берёт баг в проверку. | QA проходит шаги, проверяет регресс на соседних формах. |
| **Reopened** | Баг вернулся в работу: исправление не сработало, либо появилось новое проявление того же дефекта. | Тестировщик после неуспешного ретеста. | После фикса поле «Last Name» работает, но теперь сломалось «Postal Code» — баг переоткрыт с уточнением. |
| **Verified / Closed** | Дефект подтверждён исправленным; баг-репорт закрывается. | Тестировщик (Verified) / автоматически или релиз-менеджер (Closed после релиза). | После успешного ретеста и выпуска версии в прод баг закрывается. |
| **Rejected / Not a bug** | Описанное поведение не является дефектом (соответствует ТЗ, является фичей, либо ошибка тестировщика). | Разработчик / аналитик / PM при триаже. | «Это поведение специально для `problem_user` — учебный аккаунт». |
| **Duplicate** | Аналогичный баг уже заведён ранее. Текущий репорт связывается с оригиналом и закрывается. | Тестировщик / тимлид при триаже. | Найден более ранний баг с тем же описанием — текущий помечен Duplicate of BUG-042. |
| **Cannot Reproduce / Not Reproducible** | Дефект не удаётся воспроизвести на стенде разработчика/тестировщика. | Разработчик; обратно реоупенит тестировщик, если найдёт условия воспроизведения. | «Прошёл шаги 5 раз — всё работает. Уточните окружение». |
| **Deferred / Postponed** | Решение об исправлении принято, но работа отложена (например, до следующего релиза или до изменения соответствующего модуля). | PM / тимлид. | Баг признан, но фикс запланирован на квартал Q3 после редизайна модуля. |
| **Won't Fix / Won't Do** | Дефект признан, но исправляться не будет (например, область вне поддержки, очень редкий кейс, экономически невыгодно). | PM / Product Owner / тимлид. | Баг воспроизводится только в IE11, который мы официально не поддерживаем — Won't Fix. |
| **On Hold / Blocked** | Работа над фиксом приостановлена из-за внешней блокировки (ждём ответ от вендора, ждём фикс зависимости, ждём согласование с бизнесом). | Разработчик / PM. | Фикс зависит от обновления библиотеки X, релиз которой ожидается через 2 недели. |
| **Closed (Final)** | Финальный статус — баг полностью закрыт после выпуска фикса в прод и подтверждения отсутствия регресса. | Тестировщик / релиз-менеджер / автоматически по правилам. | Версия 1.5.0 в проде, баг проверен на проде — Closed. |

### Типовая схема переходов

```
New → Assigned → In Progress ↔ Need Info
                       ↓
                   Resolved → Ready for Test → In Verification
                                                    ↓
                                              Verified → Closed
                                                    ↓
                                                Reopened → ... (цикл)

Альтернативные ветки на любом этапе:
  → Rejected / Not a bug
  → Duplicate
  → Cannot Reproduce
  → Deferred / Won't Fix / On Hold
```

## 2. Конфликтные сценарии и стратегия коммуникации

Ниже разобраны 2 из 4 предложенных сценариев.

### Сценарий А. Разработчик говорит «Not a bug»

**Ситуация.** Я завёл баг про поле «Last Name» на чекауте. Разработчик закрыл репорт со статусом «Not a bug», аргументируя: «Это работает по дизайну для `problem_user`, ничего чинить не надо».

**Цель.** Получить либо подтверждение со ссылкой на спецификацию (тогда баг действительно закрывается с пометкой), либо переоткрыть баг и довести его до фикса. Главное — разобраться, а не «продавить».

**Пошаговая стратегия.**

1. **Не спорить эмоционально.** Сначала перечитать комментарий разработчика, перепроверить шаги и поведение ещё раз — возможно, я что-то упустил.
2. **Запросить ссылку на источник.** Попросить указать конкретный документ/тикет/ТЗ/комментарий PM, где зафиксировано «ожидаемое поведение». Если ссылки нет — это уже сигнал.
3. **Воспроизвести и зафиксировать.** Записать видео + HAR-логи, показать, что поведение блокирует основной user flow (невозможно завершить чекаут). Это трансформирует разговор из «фича/баг» в «можно ли пройти основной сценарий».
4. **Вынести вопрос на нейтральных лиц.** Подключить аналитика / PM / тимлида QA для верификации трактовки требований. Это снимает конфликт «один-на-один» и переводит в плоскость объективной оценки.
5. **Зафиксировать решение в трекере.** Любой исход (фикс, не баг, deferred) должен быть письменно прокомментирован с обоснованием — чтобы при следующих похожих кейсах было на что сослаться.

**Шаблоны коммуникации.**

> «Привет! Спасибо за быстрый ответ. Я перепроверил — поведение действительно повторяется на 5 разных билдах. Мог бы ты, пожалуйста, сослаться на тикет/спеку, где описано, что для `problem_user` это ожидаемо? Если такая спека есть — закроем как «Not a bug», но я хочу зафиксировать ссылку для команды.»

> «Прикладываю видео: пользователь физически не может пройти чекаут, и при этом форма требует заполнения поля. Мне кажется, тут есть противоречие — даже если это намеренная блокировка, как минимум, должна быть подсказка пользователю. Давай обсудим с @PM?»

> Аргументы:
>  - «Поведение блокирует ключевой бизнес-сценарий — оформление заказа.»
>  - «UX-противоречие: поле требуется, но не заполняется, без уведомления.»
>  - «Для `standard_user` поведение другое — значит, дефект сегментный и непредсказуемый.»

**Желаемый исход.** Либо ссылка на спеку (и тогда я обновлю свою тестовую модель), либо переоткрытие и фикс.

### Сценарий B. Релиз блокируется одним багом

**Ситуация.** До релиза 12 часов. Я нашёл баг (тот же про «Last Name») с серьёзностью Critical. Команда волнуется: «Без релиза не выживем, давайте катить, пофиксим в hotfix». Менеджер давит на «закрыть глаза».

**Цель.** Не позволить выкатить заведомо ломающий основной сценарий релиз; при этом не превратиться в «того самого QA, который блокирует прод», а помочь команде найти безопасный путь.

**Пошаговая стратегия.**

1. **Чётко описать impact.** Сформулировать в цифрах/процентах, какая доля пользователей/выручки/функциональности страдает. Без этого «Critical» — это пустое слово для бизнеса.
2. **Подготовить варианты решения, а не ультиматум.** QA, который только говорит «нельзя катить», — это блокатор. QA, который приходит с тремя вариантами, — это партнёр.
   - Вариант 1: feature toggle / временное отключение проблемного флоу до фикса.
   - Вариант 2: hotfix-патч в течение N часов с быстрым ретестом.
   - Вариант 3: частичный rollback изменения, вызвавшего регресс.
3. **Показать risk vs benefit.** Сопоставить риски релиза с багом и риски задержки на N часов. Бизнес поймёт «потеря 30% конверсии чекаута» лучше, чем «severity Critical».
4. **Эскалировать структурно.** Если PM настаивает на релизе, попросить письменное подтверждение — «релиз катим, я подтверждаю принятие риска» — это нормальная практика, и обычно после такого запроса бизнес сам берёт паузу.
5. **Зафиксировать решение в Confluence/тикете.** Любой исход должен быть письменным: что выкатили, какой был риск, какой план Б.
6. **Готовить план мониторинга.** Если решено катить с фикс-планом — заранее настроить алёрт на метрику «успешные чекауты», чтобы откат был быстрым.

**Шаблоны коммуникации.**

> «Команда, я понимаю важность сегодняшнего релиза. У нас на чекауте баг, который блокирует оформление заказа для X% пользователей (см. видео). До релиза 12 часов — предлагаю варианты:
>  1) hotfix в ветку release/1.5.0, ETA фикса 2 часа + 1 час ретест;
>  2) feature toggle и выпуск без проблемного флоу;
>  3) откат коммита `abc123`, который, по моей гипотезе, и привнёс регресс.
>
> Какой вариант обсуждаем? У меня готовы шаги для ретеста по любому из них.»

> «Если команда принимает решение катить с этим багом, я не блокирую, но прошу отметить в тикете, что я отметил Critical-impact и риск принят на стороне продукта. Параллельно настрою мониторинг конверсии чекаута, чтобы быстро среагировать.»

**Аргументы для PM:**
 - «Стоимость одного такого инцидента в проде ≈ X выручки за час.»
 - «Откатить релиз с проблемой потом дороже, чем задержать на 2 часа сейчас.»
 - «Поддержка получит вал жалоб — это репутация и нагрузка на людей.»

**Желаемый исход.** Команда выбирает один из безопасных вариантов; релиз либо катится с фиксом, либо переносится на короткий срок; решение и риск зафиксированы письменно.

## 3. Ролевой диалог с ИИ (отдельно к P2P-проверке)

Для подготовки к P2P-защите я планирую отыграть с ИИ сценарий А («Not a bug») в роли «несговорчивого разработчика», используя следующий промпт:

> «Сыграй роль опытного frontend-разработчика, уверенного в качестве своего кода. Ты считаешь, что мой баг про поле «Last Name» — не баг, а ожидаемое поведение для `problem_user`. Ты слегка раздражён, отвечаешь коротко и иронично, не любишь, когда QA «придумывают баги». Я начну диалог. Не соглашайся сразу — продержи позицию минимум 4 реплики, прежде чем признать аргументы, если они достаточно сильны. Мои аргументы основаны на бизнес-impact, UX-противоречиях и отсутствии письменной спеки.»

Скриншот диалога и анализ убедительности аргументов прикладываются отдельно к P2P-сессии.

**Ожидаемый вывод после симуляции (краткий черновик рефлексии):**
- Самые сильные аргументы — те, что апеллируют к бизнес-метрикам и письменным спекам.
- Самые слабые — эмоциональные («это же очевидный баг»).
- Ключевой навык — переводить разговор из «кто прав» в «давайте посмотрим, как это влияет на пользователя».
